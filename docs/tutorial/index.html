Core intuition

没有共同语义，Agent 只能“猜”对方的意思

两个 Agent 能发消息，不代表能正确协作。真正的问题通常不是传输，而是：能力是什么意思、输入要什么、输出代表什么、谁能执行哪个动作、失败后如何追溯。

  • 1A2A 负责让 Agent 发现彼此、发送任务、跟踪状态、拿到结果。
  • 2本体论 负责定义业务对象、关系、规则、动作和约束。
  • 3工程化 负责把这些定义接到数据、API、权限、验证和审计。

A2A mental model

A2A 是 Agent 间的任务协议

A2A 不要求对方暴露内部工具、prompt 或数据库。它只要求对方清楚声明自己是谁、会做什么、怎么认证、怎么接任务、怎么返回状态和结果。

Agent Card: 我是谁、会什么、怎么调用 Task: 请完成这件事 Message: 交互内容和上下文 Artifact: 交付物 Status: submitted / working / input-required / completed / failed

你要抓住的 4 个问题

  • 怎么发现 Agent：Agent Card 或 registry。
  • 怎么发起工作：Task + Message。
  • 怎么处理长任务：Streaming 或 Push Notification。
  • 怎么保证安全：认证在 HTTP 层，业务授权要自己建模。

Ontology mental model

本体论是业务世界的可执行地图

本体论不是“画一张知识图谱图”。它要回答：世界里有哪些对象，它们怎么连接，什么状态是合法的，哪些动作可以被执行，执行后世界如何变化。

  • OObject：供应商、订单、合同、客户、Agent、Skill、Task。
  • LLink：订单来自供应商，合同约束订单，Agent 执行 Skill。
  • RRule：金额超过阈值必须审批，供应商风险高不能自动下单。
  • AAction：审批、派单、生成合同、通知负责人、冻结流程。

    Engineering ladder

    从简单到复杂，不要一上来就造“大一统本体”

    最稳的路线是小步升级：先统一词，再建对象，再加关系和约束，最后接动作、Agent 和审计。

    业务词表

    先统一叫法：客户、订单、供应商、合同、风险、Agent、Skill。产物可以只是 Markdown 或 SKOS。

    对象模型

    定义 Object Type 和关键字段。不要照搬数据库表，按业务对象抽象。

    关系模型

    定义 Link Type。关系要有语义名字：供应商“履约”合同，Agent“执行”Skill。

    约束和校验

    用 JSON Schema、SHACL、数据库 constraint 或后端校验，把自然语言规则变成机器检查。

    动作和权限

    把 Action 纳入本体：谁能执行、需要哪些输入、会改变哪些对象、是否需要人工确认。

    Agent 语义目录

    把能力、输入输出、证据和限制挂到 Agent Card，形成可发现、可匹配、可审计的 A2A 目录。

    Final target

    最终目标：Agent to Agent 的语义目录

    目录不只是“有哪些 Agent”。它应该能回答：这个 Agent 的能力属于哪个业务本体，输入输出能否验证，调用者有没有权限，执行后产生什么证据。

    Agent Card metadata.ontologyProfileUrl skills[].semanticType skills[].inputShapeUrl skills[].outputShapeUrl skills[].actionPolicyUrl Semantic Profile Object / Link / Logic / Action version / provenance / evaluation evidence

    落地时先做这 3 件事

    • 1给一个 Agent Card 增加 `ontologyProfileUrl`，先外链，不改 A2A 主协议。
    • 2用 schema/shape 验证一个 skill 的输入输出，先挡住错误调用。
    • 3把一次 Task 执行记录成事件图，能查谁调用、基于什么、产出什么。
