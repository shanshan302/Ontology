## A2A × 本体论交叉洞察

- **分析判断 | Agent Card 可以成为“轻量能力目录”，但不是完整本体。** 官方 [Protocol Specification](https://a2a-protocol.org/latest/specification/) 与 [Definition](https://a2a-protocol.org/latest/definitions/) 中 Agent Card 描述 identity、capabilities、skills、interfaces、security；本体层可补充 skill 的领域类、输入输出类型、前置条件、后置条件、数据权限、风险等级和 SLA。建议采用 Agent Card metadata/extension 指向外部 RDF/JSON-LD/SHACL 描述，而不是把复杂本体直接塞进 card 主体。
- **分析判断 | 多租户 A2A 路由需要本体级权限模型。** A2A tenant/interface 解决“请求去哪”，securitySchemes 解决“谁能请求”；本体层应解决“该主体能看哪些对象、能触发哪些动作、动作改变哪些业务状态”。这与 Palantir 风格 Object/Link/Logic/Action 模型天然相邻。
- **分析判断 | 任务历史是语义互操作的关键资产。** A2A issue 中的 coherent task history、multi-turn semantics、streaming 讨论说明协议层仍在定义“长任务如何被理解”。如果任务历史能映射到 RDF/事件本体，就可以支持跨 Agent 的审计、重放、归因、异常检测和合规报告。
- **分析判断 | 能力声明需要从“自报能力”升级到“可验证能力”。** 近期 A2A 相关论文如 [Capability Advertisement as a Market for Lemons](https://arxiv.org/abs/2606.03034) 指出 Agent 能力广告可能不真实或随模型漂移；本体 + 测试集 + provenance 可构成 capability proof：声明、样例、边界条件、失败率和版本一起发布。
- **分析判断 | A2A 与 MCP 的语义分工建议：MCP 工具 schema 约束局部调用，A2A 本体约束跨组织协作。** MCP 负责单 Agent 内部工具/数据接口，A2A 负责 Agent 间 task/message；本体层应跨两者保持统一 ID、对象类型、动作语义和策略标签。

## 可执行建议

1. **为 A2A Agent Card 设计最小语义扩展草案。** 字段建议包括 `domainOntologyUrl`、`skillSemanticType`、`inputShape`、`outputShape`、`preconditions`、`postconditions`、`riskClass`、`provenancePolicy`，外链 JSON-LD/RDF/SHACL，不破坏 A2A 主 schema。
2. **基于 a2a-samples 的多租户样例验证“tenant + ontology policy”。** 将 URL 子路径租户、认证主体、对象权限、动作权限映射到一份可测试策略表，重点验证跨租户 Agent Card 泄漏和 task history 隔离。
3. **把 nano-ontoprompt 作为本体侧实验底座。** 先不做全量集成，优先产出一个从 Object/Link/Logic/Action 导出 Agent skill semantic profile 的脚本或文档。
4. **跟进 A2A CLI 提案。** 若官方 CLI 成形，优先贡献/封装 `agent-card validate`、`semantic-profile validate`、`task-history export` 三类命令，便于 CI 和企业验收。
5. **将任务历史建模为事件流。** 参考 VQA execution records 与 FOI-O，把 A2A Task/Message/Artifact/StatusUpdate 映射到可查询事件本体，并用 SHACL 验证关键字段完整性。
